# Step7C-local-iter-0 Bounded Local-Lane Iteration Sidecar

## Scope

Step7C-local-iter-0 is a sidecar-only thin sequential validation loop. It does **not** integrate with the contest runtime solver, does **not** change finalizer semantics, does **not** start broad multi-iteration optimization, and does **not** use RL or trained ML.

This step follows Step7C-real-F's decision:

```text
promote_to_step7c_local_lane_iteration
```

The goal is to test whether dominance-aware local slack-fit wins survive when candidates are regenerated from updated real placement states rather than replayed from the Step7C-real-E artifact pool.

## Implementation

Implemented in `src/puzzleplace/search/local_lane_iteration.py` and run by `scripts/step7c_local_iter0_run_bounded_local_lane_iteration.py`.

For each case and each bounded iteration:

1. Start from the current placement state.
2. Regenerate the Step7C-real-E local slack-fit / HPWL-directed candidate families from that current state.
3. Evaluate each candidate with Step7G locality routing, official-like hard checks, and official-like metric deltas against the current state.
4. Apply Step7C-real-F dominance-aware preselection.
5. Select at most one current-Pareto official-like improving local edit.
6. Apply the selected edit to the current state.
7. Recompute cumulative metrics against the original layout.
8. Stop the case early when no current-Pareto acceptable local edit exists.

Regional, macro, and global lanes are not mixed into this local iteration. Global candidates remain report-only and are never sent to bounded-local repair.

## Interpretation

The F-stage replay signal partially survives real sequential application: two cases accept official-like improving local edits, both remain hard feasible, no global move is routed to bounded-local repair, and cumulative metrics do not regress after applying selected edits.

However, this is not yet strong enough for broad local-lane search. Only 2 of the 16 F-stage one-step official-like wins reproduce as selected sequential edits under the current strict current-Pareto policy. Most cases stop at iteration 0 because regenerated candidates are feasible but not official-like improving; one selected case also starves after updating the state.

The bottleneck is now sequential preselection/selection under changed state, not route safety or hard legality.

## Decision

```text
refine_sequential_local_preselection
```

Reason: sequential local edits produce real cumulative official-like improvements without feasibility collapse, but the improvement coverage is too narrow to promote to broader Step7C local-lane search.

## Next implementation direction

Recommended next step:

1. Keep the bounded local-lane sidecar and Step7G safety layer.
2. Refine sequential preselection to avoid starving immediately after the first state update.
3. Compare current-Pareto selection against a small original-Pareto/current-Pareto hybrid policy.
4. Continue reporting filtered candidates and no-improvement cases; do not silently discard them.
5. Do not pivot to coarse region planning until the sequential local policy is refined once more.
